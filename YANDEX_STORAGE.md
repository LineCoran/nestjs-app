# Подключение Yandex Object Storage для загрузки картинок

Загруженные через `POST /api/admin/upload` картинки хранятся не на диске сервера,
а в облаке (S3-совместимое **Yandex Object Storage**). Благодаря этому файлы
**переживают пересборку и переезд сервера** и раздаются по постоянным публичным ссылкам.

> Если переменные `S3_*` в `.env` **не заданы** — приложение пишет файлы на локальный
> диск в папку `uploads/` (удобно для локальной разработки). Облако нужно только для прода.

---

## Шаг 1. Создать каталог и платёжный аккаунт

1. Зайти в [Yandex Cloud Console](https://console.cloud.yandex.ru/).
2. Если ещё нет — создать **облако** и **каталог** (folder).
3. Привязать платёжный аккаунт (Object Storage платный, но для картинок сайта
   стоит копейки; есть грант для новых аккаунтов).

---

## Шаг 2. Создать бакет (bucket)

1. В консоли открыть **Object Storage** → **Создать бакет**.
2. Задать **имя** бакета (например `gory-uploads`). Имя глобально уникально и
   станет частью публичного URL — латиница, цифры, дефисы.
3. **Класс хранилища**: `Стандартное`.
4. **Доступ на чтение объектов**: выбрать **`Публичный`** (`Public`).
   Это позволит отдавать картинки по прямой ссылке без подписи.
   > Доступ на запись и листинг оставить `Ограниченный`.
5. Создать.

После создания публичный адрес объектов будет такого вида:

```
https://storage.yandexcloud.net/<имя-бакета>/<имя-файла>
```

---

## Шаг 3. Создать сервисный аккаунт и выдать права

1. В каталоге открыть **Identity and Access Management (IAM)** →
   **Сервисные аккаунты** → **Создать**.
2. Имя — например `gory-storage`.
3. Назначить роль **`storage.editor`** (чтение + запись объектов).
   > Минимально достаточно `storage.uploader` для записи, но `storage.editor`
   > удобнее (позволит и удалять файлы при необходимости).

---

## Шаг 4. Выпустить статический ключ доступа (Access Key)

1. Открыть созданный сервисный аккаунт → вкладка **Ключи доступа** /
   **Создать новый ключ** → **Статический ключ доступа**.
2. Скопировать оба значения:
   - **Идентификатор ключа** → это `S3_ACCESS_KEY_ID`
   - **Секретный ключ** → это `S3_SECRET_ACCESS_KEY`
     (показывается **один раз** — сохрани сразу).

> ⚠️ Это секреты. Их место — только в `.env` на сервере. Никогда не коммить
> в репозиторий и не вставляй в код. Это **не** OAuth-токен Яндекса и **не**
> ключ от Яндекс.Диска — нужен именно «статический ключ доступа» из Object Storage.

---

## Шаг 5. Прописать переменные в `.env`

На сервере в файле `.env` (рядом с приложением) заполнить блок:

```dotenv
S3_ENDPOINT=https://storage.yandexcloud.net
S3_REGION=ru-central1
S3_BUCKET=gory-uploads
S3_ACCESS_KEY_ID=<идентификатор ключа из шага 4>
S3_SECRET_ACCESS_KEY=<секретный ключ из шага 4>
S3_PUBLIC_URL=https://storage.yandexcloud.net/gory-uploads
S3_FORCE_PATH_STYLE=true
S3_ACL=
```

Пояснения:

| Переменная             | Значение                                                                 |
| ---------------------- | ------------------------------------------------------------------------ |
| `S3_ENDPOINT`          | Точка входа Yandex Object Storage (всегда одна и та же).                  |
| `S3_REGION`            | Для Yandex — `ru-central1`.                                               |
| `S3_BUCKET`            | Имя бакета из шага 2.                                                     |
| `S3_ACCESS_KEY_ID`     | Идентификатор статического ключа.                                        |
| `S3_SECRET_ACCESS_KEY` | Секретный ключ.                                                          |
| `S3_PUBLIC_URL`        | База для ссылок в ответе ручки. Обычно `endpoint/bucket` или свой CDN-домен. |
| `S3_FORCE_PATH_STYLE`  | `true` для Yandex (адресация вида `endpoint/bucket/key`).                 |
| `S3_ACL`               | При **публичном** бакете оставить **пустым**. Для приватного — `public-read`. |

---

## Шаг 6. Перезапустить API и проверить

1. Перезапустить приложение (`npm run start:prod` / systemd / pm2 — как у тебя поднято).
2. В логах при старте должно появиться:

   ```
   Хранилище: S3 (https://storage.yandexcloud.net, bucket=gory-uploads)
   ```

   Если видишь `Хранилище: локальный диск (S3_* не заданы)` — значит одна из
   переменных не подхватилась (проверь, что `.env` читается и заполнены все четыре:
   `S3_ENDPOINT`, `S3_BUCKET`, `S3_ACCESS_KEY_ID`, `S3_SECRET_ACCESS_KEY`).

3. Загрузить тестовую картинку:

   ```bash
   curl -X POST https://<домен>/api/admin/upload \
     -H "Authorization: Bearer <JWT админа>" \
     -F "file=@/path/to/photo.jpg"
   ```

   Ответ:

   ```json
   {
     "url": "https://storage.yandexcloud.net/gory-uploads/<uuid>.jpg",
     "filename": "<uuid>.jpg",
     "size": 123456
   }
   ```

4. Открыть `url` из ответа в браузере — картинка должна отдаться напрямую из облака.

## Перенос старых картинок (если есть)

Если на сервере остались файлы в папке `uploads/` со старыми ссылками `/uploads/...`,
их можно разово залить в бакет, чтобы не потерять. Удобнее всего через
[`ya-cli`/`aws cli`](https://yandex.cloud/ru/docs/storage/tools/aws-cli) или `rclone`:

```bash
# пример через aws cli (endpoint Яндекса)
aws --endpoint-url=https://storage.yandexcloud.net \
  s3 cp ./uploads/ s3://gory-uploads/ --recursive
```

После этого при необходимости обновить старые `/uploads/...` ссылки в БД на
`https://storage.yandexcloud.net/gory-uploads/...`.

## Частые ошибки

- **403 / AccessDenied при открытии ссылки** — у бакета не включён публичный доступ
  на чтение объектов (шаг 2.4) либо неверный `S3_PUBLIC_URL`.
- **403 при загрузке (`curl`)** — у сервисного аккаунта нет роли `storage.editor`,
  либо неверные `S3_ACCESS_KEY_ID` / `S3_SECRET_ACCESS_KEY`.
- **SignatureDoesNotMatch** — обычно неверный `S3_REGION` (должен быть `ru-central1`)
  или повреждённый секретный ключ.
- **Пишет на локальный диск, а не в облако** — не заполнена хотя бы одна из
  обязательных переменных; приложение в этом случае молча падает в фолбэк.
