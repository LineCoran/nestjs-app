# Техническое задание: сайт бронирования туров «Горы по колено»

## 1. Контекст и стек

Проект уже начат:
- **Frontend**: Vue 3 + TypeScript
- **Backend**: NestJS
- **БД**: PostgreSQL (через Docker-контейнер)

Задача — доработать backend (NestJS + PostgreSQL) и frontend (Vue + TS) в соответствии с описанной ниже структурой данных, страницами и функциональностью. Документ описывает сущности, API, страницы и экраны админки. Конкретные UI-компоненты, схему БД (миграции), DTO и структуру модулей AI-агент проектирует самостоятельно на основе данного описания.

---

## 2. Состав проекта

1. **Публичный сайт (frontend)**:
    - Главная страница
    - Страница каталога туров (со списком и фильтрами)
    - Страница отдельного тура
    - Страницы «Блог» (список + статья)
    - Страница «Команда гидов»
    - Страница «О нас» / «Компания»
2. **Backend (API)** на NestJS, обслуживающий публичный сайт и админку
3. **Админ-панель (frontend)**: защищена авторизацией, позволяет управлять турами, блогом, командой гидов, контентом страницы «О нас» и просматривать заявки

## 3. Сущности данных (Entities)

### 3.1 Tour (Тур)
Основная сущность. Поля:
- `id`
- `slug` — для URL (`/tours/tolbachik-po-sledam-izverzheniy`)
- `title` — название тура
- `subtitle` / `description` — краткое описание (вступительный абзац)
- `category` — категория/направление (для фильтров: «Вулканы», «Хели-ски», «Лето», «Сплав» и т.д.) — см. раздел 3.2
- `coverImage` — главное изображение (обложка для карточки и шапки страницы)
- `gallery` — массив изображений (галерея на странице тура)
- `duration` — продолжительность в днях (например, «7 дней»)
- `groupSize` — размер группы (например, «8-12 чел.»)
- `difficulty` — сложность (enum: лёгкая / средняя / сложная)
- `season` — сезон проведения (например, «июн-сен»)
- `nearestDate` — дата ближайшего тура (для блока «Близжайший тур»)
- `badges` — массив меток-бейджей (например, «новинка», «хит», «открыто бронирование на 2027»)
- `aboutText` — расширенное текстовое описание («О туре», поддержка многоабзацного текста / rich text)
- `program` — программа тура: массив дней/этапов (см. 3.3 TourProgramItem)
- `included` — список «Что входит в стоимость» (массив строк)
- `excluded` — список «Что не входит» (массив строк)
- `whatToTake` — список категорий «Что взять с собой» (например, «Личные вещи», «Снаряжение»), каждая со своим списком пунктов
- `importantInfo` — блоки «Важно знать» (например, «Погодные условия», «Безопасность», «Скидка», «Осталось мест») — массив объектов {icon, title, description, type (info/warning/success)}
- `priceOptions` — варианты цены/формата тура (см. 3.4 TourPriceOption)
- `relatedTours` — связанные/рекомендуемые туры (массив ссылок на другие туры, для блока «С туром чаще выбирают»)
- `isPublished` — флаг публикации (черновик/опубликован)
- `createdAt`, `updatedAt`

### 3.2 TourCategory (Категория тура)
Используется для фильтров на странице каталога («Популярные», «Хели-ски», «Лето», «Вулканы», «Сплав» и т.д.)
- `id`
- `name`
- `slug`
- `toursCount` — вычисляемое поле (опционально, можно считать на лету)

### 3.3 TourProgramItem (Пункт программы тура)
Вложенная сущность (связь один-ко-многим с Tour), порядок задаётся полем `order`.
- `id`
- `tourId`
- `order` — порядковый номер (для отображения «1», «2» и сортировки)
- `title` — название этапа («Сбор участников», «Восхождение на Плоский Толбачик» и т.д.)
- `description` — текст описания этапа (опционально, может быть пустым — тогда отображается только заголовок)
- `tags` — массив тегов-плашек (например, «Акклиматизация», «Завтрак», «Ужин»)

### 3.4 TourPriceOption (Вариант цены/формата тура)
Вложенная сущность (связь один-ко-многим с Tour). У одного тура может быть несколько форматов (например, «Вахтовый автобус», «Джипы», «Вертолёт»), у каждого — своя цена и доступные даты.
- `id`
- `tourId`
- `formatName` — название формата («Вахтовый автобус», «Джипы», «Вертолёт»)
- `priceFrom` — цена «от» за человека
- `maxGroupSize` — максимальный размер группы для этого формата
- `availableDates` — массив доступных дат (см. 3.5 TourAvailableDate)

### 3.5 TourAvailableDate (Доступная дата)
Вложенная сущность (связь один-ко-многим с TourPriceOption).
- `id`
- `priceOptionId`
- `dateFrom`, `dateTo` — диапазон дат заезда
- `availability` — статус доступности (enum: `available` — 3+ места / `limited` — 1-2 места / `full` — мест нет)

### 3.6 Booking (Заявка / Lead)
Единая сущность для всех форм на сайте (на странице тура, на главной, форма «индивидуального» подбора).
- `id`
- `name` — имя клиента
- `phone` — телефон
- `contactMethod` — предпочтительный способ связи (телефон/телеграм/whatsapp и т.д., опционально)
- `source` — источник заявки (enum: `tour_page` — со страницы тура, `homepage` — главная страница, `custom` — индивидуальный подбор)
- `tourId` — ссылка на тур (опционально, может быть пустой для индивидуальной заявки)
- `priceOptionId` — выбранный формат тура (опционально)
- `desiredDates` — желаемые даты (текст или диапазон дат)
- `peopleCount` — количество человек
- `tourFormat` — формат тура текстом (для случаев, когда выбор шёл текстом, например «Джип-тур»)
- `comment` — комментарий/пожелания клиента
- `preferences` — массив предпочтений по туру (для формы «индивидуально»: «Медведи», «Восхождения», «Джипы», «Сплавы», «Вулканы», «Фрирайд», «Долина гейзеров» и т.д.)
- `isCustomRequest` — флаг «Хочу индивидуально»
- `totalPrice` — расчётная итоговая стоимость (если применимо)
- `status` — статус обработки заявки (enum: `new`, `in_progress`, `done`, `rejected`) — для удобства работы менеджера в админке
- `createdAt`

### 3.7 BlogPost (Статья блога)
- `id`
- `slug`
- `title`
- `excerpt` — краткое описание/анонс
- `coverImage`
- `content` — текст статьи (rich text / markdown)
- `isPublished`
- `publishedAt`
- `createdAt`, `updatedAt`

### 3.8 Guide (Гид/сотрудник команды)
Для раздела «Познакомьтесь с нашими гидами».
- `id`
- `name`
- `role` — должность («Инструктор, проводник», «Водитель» и т.д.)
- `photo`
- `description` — краткое описание/биография
- `order` — порядок отображения

### 3.9 CompanyInfo (Контент страницы «О нас» / блока «О нас» на главной)
Единственная запись (singleton), редактируемая через админку.
- `id`
- `aboutText` — текст «О нас» (rich text, поддержка нескольких абзацев)
- `aboutImage` — фото для блока «О нас»
- `stats` — массив статистик-бейджей («6+ лет опыта», «2,5+ тысяч туристов», «4.9 рейтинг», «МЧС сертификаты», «РТО лицензия») — каждый со своим значением и подписью
- `heroTitle`, `heroSubtitle`, `heroImage` — контент главного экрана (hero) главной страницы
- `contactPhone`, `telegramLink`, `vkLink` — контактные данные для шапки/футера

### 3.10 AdminUser (Администратор)
- `id`
- `email`
- `passwordHash`
- `createdAt`

## 4. Хранение изображений

Поскольку проект на NestJS + PostgreSQL без облачного хранилища — рекомендуется:
- Хранить файлы изображений (обложки туров, галереи, фото гидов и т.д.) на файловой системе backend-сервера в отдельной директории (например, `/uploads`), раздавая их через статический маршрут NestJS (`ServeStaticModule`)
- В БД хранить только относительный путь/URL к файлу (строка), а не сами бинарные данные
- Реализовать отдельный эндпоинт загрузки файлов (`POST /api/admin/upload`), принимающий `multipart/form-data`, сохраняющий файл с уникальным именем (uuid) и возвращающий URL
- Для туров с галереей (`gallery`) хранить массив URL (тип `text[]` или отдельная таблица `tour_images` с полем `order`)
- Ограничить допустимые форматы (jpg, png, webp) и максимальный размер файла на бэкенде

*Примечание:* в дальнейшем (вне рамок MVP) можно перейти на S3-совместимое хранилище (например, MinIO), заменив реализацию сервиса загрузки без изменения остальной архитектуры.

## 5. Публичные страницы (Frontend)

### 5.1 Главная страница (`/`)
- Hero-блок: заголовок, подзаголовок, фоновое изображение, кнопки «Забронировать тур» / «Направления» (контент из `CompanyInfo`)
- Блок «Выберите своё приключение» — каталог туров с фильтрами по категориям (горизонтальный список тегов-категорий), сетка карточек туров (превью: обложка, бейджи, название, цена «от»), кнопка «Посмотреть все» → переход на страницу каталога
- Блок «О нас» — текст + фото (из `CompanyInfo`)
- Блок статистики компании (бейджи: годы опыта, туристы, рейтинг, сертификаты)
- Блок «Команда гидов» — карточки гидов (фото, имя, должность, описание)
- Блок «Готовы увидеть Камчатку?» — баннер с кнопками
- Блок «Блог» — карусель/список последних статей
- Блок формы заявки «Не знаете что выбрать?» — форма индивидуального подбора (имя, телефон, желаемая дата, способ связи, чекбокс «хочу индивидуально», комментарий, теги предпочтений) → создаёт `Booking` с `source = custom`, `isCustomRequest = true`

### 5.2 Каталог туров (`/tours`)
- Список всех туров с фильтрацией по категориям (`TourCategory`)
- Карточки туров аналогичны блоку на главной
- Пагинация или подгрузка по скроллу (на усмотрение AI-агента)

### 5.3 Страница тура (`/tours/:slug`)
- Хлебные крошки
- Заголовок, краткое описание
- Блок характеристик (продолжительность, группа, сложность, сезон, ближайшая дата)
- Галерея изображений (с возможностью открыть лайтбокс)
- Блок «О туре» — текст
- «Программа тура» — аккордеон по `TourProgramItem` (с порядковыми номерами, заголовками, описанием, тегами)
- «Цена тура» — табы по `TourPriceOption` (формат тура), внутри — цена, доступные даты с индикацией мест (`TourAvailableDate`), кнопки «Написать ВКонтакте» / «Забронировать»
- «Что входит в стоимость» / «Что не входит»
- «Что взять с собой» — аккордеон по категориям
- «Важно знать» — информационные блоки
- «С туром чаще выбирают» — карусель связанных туров (`relatedTours`)
- Форма заявки «Приезжайте на Камчатку» — поля: имя, телефон, желаемые даты, способ связи, формат тура, количество человек (с автоматическим расчётом «Итого» на основе `priceFrom * peopleCount`), комментарий, согласие на обработку данных → создаёт `Booking` с `source = tour_page`, `tourId`, `priceOptionId`
- Виджет бронирования в шапке (sticky-блок справа, как на скриншоте) с теми же полями — может использовать ту же форму/компонент

### 5.4 Блог (`/blog`, `/blog/:slug`)
- Список статей с превью (обложка, заголовок, краткое описание)
- Страница отдельной статьи с полным текстом

### 5.5 Команда гидов (`/team` или раздел в составе «Компания»)
- Список карточек гидов

### 5.6 О компании (`/company`)
- Текст «О нас», фото, статистика (из `CompanyInfo`)
- Может включать блок команды гидов

## 6. Backend API (основные группы эндпоинтов)

### 6.1 Публичные эндпоинты (без авторизации)
- `GET /api/tours` — список туров с фильтрацией по категории (`?category=slug`), пагинацией
- `GET /api/tours/:slug` — детали тура со всеми связанными данными (программа, варианты цен, даты, related tours)
- `GET /api/tour-categories` — список категорий
- `GET /api/blog` — список опубликованных статей
- `GET /api/blog/:slug` — статья
- `GET /api/guides` — список гидов
- `GET /api/company-info` — контент главной/«О нас»
- `POST /api/bookings` — создание заявки (публичная форма)

### 6.2 Авторизация
- `POST /api/auth/login` — вход администратора (email + пароль), возвращает JWT
- JWT-защита всех `/api/admin/*` маршрутов (Guard в NestJS)
- Один тип роли — `admin`, без разграничения прав

### 6.3 Админские эндпоинты (требуют JWT)
- `GET/POST/PATCH/DELETE /api/admin/tours` — CRUD туров (включая вложенные program items, price options, available dates)
- `GET/POST/PATCH/DELETE /api/admin/tour-categories` — CRUD категорий
- `GET/POST/PATCH/DELETE /api/admin/blog` — CRUD статей блога
- `GET/POST/PATCH/DELETE /api/admin/guides` — CRUD гидов
- `GET/PATCH /api/admin/company-info` — редактирование singleton-контента
- `GET /api/admin/bookings` — список заявок с фильтрацией по статусу/источнику/дате
- `PATCH /api/admin/bookings/:id` — изменение статуса заявки
- `POST /api/admin/upload` — загрузка изображений

## 7. Админ-панель (Frontend)

### 7.1 Авторизация
- Страница логина (`/admin/login`) — email + пароль
- После входа — редирект в дашборд админки, JWT хранится (например, в httpOnly cookie или localStorage — на усмотрение AI-агента)
- Защита всех маршрутов `/admin/*` через router guard

### 7.2 Разделы админки
1. **Заявки (Bookings)**
    - Таблица заявок: имя, телефон, источник, тур (если есть), дата создания, статус
    - Фильтры по статусу и источнику
    - Просмотр деталей заявки (все поля), изменение статуса
2. **Туры (Tours)**
    - Список туров (таблица с поиском, фильтром по категории, статусом публикации)
    - Форма создания/редактирования тура:
        - Основные поля (название, slug, описание, категория, характеристики)
        - Загрузка обложки и галереи изображений
        - Редактор программы тура (добавление/удаление/сортировка этапов с тегами)
        - Редактор вариантов цены (добавление форматов, для каждого — даты с указанием доступности)
        - Списки «что входит/не входит», «что взять с собой», «важно знать»
        - Выбор связанных туров
        - Переключатель публикации
3. **Категории туров** — CRUD простого списка (название, slug)
4. **Блог**
    - Список статей, CRUD, редактор текста (rich text), загрузка обложки, переключатель публикации
5. **Команда гидов**
    - Список гидов, CRUD (имя, должность, фото, описание, порядок)
6. **О компании**
    - Форма редактирования singleton-контента: hero-блок главной, текст «О нас», статистика, контакты

## 8. Что не входит в MVP

- Онлайн-оплата (вопрос остаётся открытым; в дальнейшем может быть добавлена интеграция с платёжным провайдером для `Booking`)
- Многоролевая модель доступа в админке
- Личный кабинет пользователя сайта
- Уведомления (email/телеграм-бот) о новых заявках — может быть добавлено отдельным этапом

## 9. Дополнительные технические рекомендации

- Все текстовые поля с поддержкой форматирования (`aboutText`, `content` статей блога, описание тура) — хранить как HTML или Markdown, на frontend использовать соответствующий рендерер/редактор (например, TipTap для Vue)
- Слаги (`slug`) генерировать автоматически из названия с возможностью ручного редактирования, проверять уникальность
- Для расчёта «Итого» в форме бронирования — использовать `priceFrom` выбранного `TourPriceOption * peopleCount` на frontend, опционально пересчитывать и валидировать на backend при создании `Booking`
- Валидация форм — на frontend (UX) и backend (DTO с class-validator в NestJS)
- Все админские операции CRUD — стандартные REST-эндпоинты с пагинацией для списков
